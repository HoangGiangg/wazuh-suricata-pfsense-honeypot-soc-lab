\# Wazuh Custom Detection Rules



\## Overview



This directory contains custom Wazuh detection rules developed for the SOC lab.



The rules are designed to detect suspicious activity from Honeypot services, Suricata IDS alerts, Server-real host events, and AI-generated risk alerts.



\## Rule Files



| File | Purpose |

|---|---|

| honeypot\_rules.xml | Detects Cowrie, Heralding, and Snare/Tanner Honeypot activity |

| suricata\_correlation\_rules.xml | Detects and correlates Suricata alerts collected from pfSense |

| server\_real\_rules.xml | Detects suspicious activity on the protected Server-real |

| ai\_alert\_rules.xml | Detects AI-generated alert risk levels |



\## Rule Categories



\### Honeypot Rules



The Honeypot rules detect activity from:



\- Cowrie SSH Honeypot

\- Heralding FTP, SMTP, POP3, and IMAP Honeypot

\- Snare/Tanner Web Honeypot



Detection examples:



\- SSH connection to Cowrie

\- SSH login failure

\- SSH brute-force behavior

\- Successful fake login

\- Command execution inside Cowrie

\- FTP or mail service credential attack

\- Suspicious web request

\- SQL Injection attempt

\- XSS attempt

\- Path Traversal or LFI attempt

\- Malware download behavior



\### Suricata Correlation Rules



The Suricata correlation rules process IDS alerts generated by Suricata on pfSense.



Detection examples:



\- ICMP probing

\- Port scanning

\- Reconnaissance activity

\- Web attack signature

\- High-severity IDS alert

\- Dropped or blocked traffic

\- Repeated scan activity

\- Redirect candidate based on Suricata alerts



\### Server-real Rules



The Server-real rules detect suspicious host-based activity on the protected server.



Detection examples:



\- SSH authentication failure

\- SSH brute-force behavior

\- Successful sudo usage

\- Sensitive sudo command execution

\- Web probing against protected web service

\- Repeated web probing

\- Nginx configuration change

\- Web content modification

\- Privilege escalation followed by configuration change

\- Web probing followed by web content modification



\### AI Alert Rules



The AI alert rules classify AI-generated alerts by predicted risk level.



Detection examples:



\- Medium risk AI alert

\- High risk AI alert

\- Critical risk AI alert



\## Rule ID Ranges



| Range | Purpose |

|---|---|

| 100100 - 100199 | Honeypot detection rules |

| 100200 - 100299 | Server-real host-based detection rules |

| 100500 - 100599 | Suricata and redirect candidate rules |

| 101800 - 101899 | AI-generated alert rules |



\## Deployment Notes



These rules should be placed in the Wazuh manager custom rules directory.



Example location:



/var/ossec/etc/rules/



After updating rules, restart the Wazuh manager.



Example command:



systemctl restart wazuh-manager



\## Security Notes



\- Review all rules before deploying them in a production environment.

\- Tune frequency and timeframe values to reduce false positives.

\- Do not upload sensitive logs, private IPs from real environments, passwords, tokens, or raw production alerts.

\- This ruleset is designed for an isolated lab environment.
